# ADR 0053: 공개 목록 응답에 빈 결과 신호 추가

- 상태: 승인
- 날짜: 2026-06-20
- 범위: 공지·학사 정책 검색·캠퍼스 시설 공개 MCP 응답

## 배경

공개 조회 도구가 정상 실행됐지만 조건에 맞는 항목이 없으면 기존 응답에는 빈 배열만 남았다. 사람은 빈 배열을 보고 “검색 결과 없음”으로 해석할 수 있지만, LLM 클라이언트는 이를 응답 누락, 파싱 실패, 추가 페이지 존재 또는 도구 실패와 혼동할 수 있다. 특히 공지 응답은 페이지 메타데이터가 함께 있고, 학사 정책 검색 응답은 검색 근거인 `evidence`와 검색에 사용한 코퍼스 출처인 `sources`가 함께 있어 어떤 목록이 실제 검색 결과인지 명시할 필요가 있다.

MCP 도구 명세는 도구 결과를 `structuredContent` JSON 객체로 제공하고 `outputSchema`로 검증할 수 있도록 정의한다. 따라서 빈 결과 의미도 자연어 추론에 맡기기보다 구조화 필드로 명시하는 편이 클라이언트가 안정적으로 분기하기 쉽다.

## 검토한 대안

### 대안 1: 빈 배열만 유지하고 도구 설명을 보강

- 장점: 응답 스키마 변경이 없다.
- 기각 이유: 호출 시점의 실제 상태가 응답 데이터에 없으므로 클라이언트가 매번 도구별 주 목록을 찾아 길이를 검사해야 한다. 안내 문구도 런타임 결과에 포함되지 않아 LLM이 빈 결과를 실패로 오해하는 문제를 직접 해결하지 못한다.

### 대안 2: 기존 목록 응답을 공통 envelope로 교체

- 장점: 모든 공개 목록 도구에 동일한 메타데이터 구조를 적용할 수 있다.
- 기각 이유: 기존 필드를 이동하거나 최상위 JSON 형태를 바꾸므로 외부 MCP 클라이언트와 위치 기반 파서에 breaking change가 된다. `List<...>`를 직접 반환하는 `list_academic_policy_sources`, `get_academic_calendar`, `find_academic_calendar_events`도 객체로 감싸야 하므로 이번 additive-only 범위를 벗어난다.

### 대안 3: 기존 wrapper의 마지막에 `empty`, `note`를 추가

- 장점: 기존 필드 이름·타입·선언 순서를 그대로 유지하면서 빈 결과를 기계 판독값과 사용자 안내 문구로 동시에 표현한다. raw `List` 도구의 JSON 형태도 건드리지 않는다.
- 단점: wrapper별 팩토리와 빈 결과 문구가 필요하다.
- 선택: 기존 클라이언트 호환성을 지키면서 문제를 직접 해결할 수 있어 채택했다.

## 결정

다음 세 wrapper record의 기존 컴포넌트 뒤에 `boolean empty`, `String note`를 순서대로 추가한다.

| 응답 | 주 목록 | 빈 결과 안내 |
| --- | --- | --- |
| `NoticeListResponse` | `items` | `조건에 맞는 공지가 없어요.` |
| `AcademicPolicySearchResponse` | `evidence` | `관련 공식 출처를 찾지 못했어요.` |
| `CampusFacilityListResponse` | `facilities` | `검색 조건에 맞는 시설이 없어요.` |

`empty`는 주 목록이 `null`이거나 비어 있을 때만 `true`다. `note`는 `empty=true`일 때만 위 한국어 문구를 가지며, 결과가 있으면 `null`이다. 학사 정책 검색은 `totalMatches`와 대응하는 실제 검색 결과인 `evidence`를 기준으로 한다. `sources`는 검색에 사용한 공식 코퍼스 목록이므로 출처가 존재하더라도 근거가 검색되지 않은 상태를 `empty=true`로 표현한다.

각 record는 기존 필드 인자를 받는 `of(...)` 정적 팩토리에서 두 신규 필드를 계산한다. 서비스, connector, 테스트의 모든 생성 지점은 이 팩토리를 사용해 동일한 규칙을 공유한다.

## additive-only 호환성

- 기존 JSON 필드의 이름, 타입, 선언 순서를 변경하지 않는다.
- 신규 필드는 record 컴포넌트의 마지막에만 추가한다.
- 결과가 있는 응답의 기존 필드 값은 그대로이며 `empty:false`, `note:null`만 뒤에 추가된다.
- `NoticeCompactListResponse`와 `Object` 반환 도구의 분기 구조는 변경하지 않는다.
- raw `List<...>` 반환 도구는 wrapper 도입 자체가 최상위 JSON 형태 변경이므로 이번 범위에서 제외한다.

## 동작과 검증

1. 서비스 또는 connector가 기존과 동일하게 목록과 메타데이터를 만든다.
2. 응답 record의 `of(...)` 팩토리가 주 목록의 `null`/empty 여부를 계산한다.
3. Jackson은 기존 컴포넌트 뒤에 `empty`, `note`를 직렬화한다.
4. LLM 클라이언트는 배열 구조를 도구별로 추론하지 않고 `empty`로 분기하고, 필요하면 `note`를 사용자에게 전달한다.
5. 테스트는 세 응답 각각에 대해 결과 있음(`empty=false`, `note=null`)과 결과 없음(`empty=true`, 한국어 안내)을 검증한다. REST 직렬화 테스트는 신규 필드가 실제 JSON에 포함됨을 확인한다.

## 포트폴리오 가치

단순한 문구 추가가 아니라, 구조화 도구 응답에서 “성공했지만 결과 없음”과 “도구 실패”를 분리하고 외부 스키마 호환성을 유지한 사례다. wrapper 응답과 raw 배열 응답의 변경 위험을 구분하고, 검색 결과 목록과 코퍼스 목록의 의미 차이까지 명시적으로 모델링했다는 점을 설명할 수 있다.

## 예상 면접 질문

1. 빈 배열만으로 충분하지 않고 `empty`와 `note`를 함께 둔 이유는 무엇인가요?
2. 학사 정책 검색에서 `sources`가 아니라 `evidence`를 빈 결과 판정 기준으로 선택한 이유는 무엇인가요?
3. raw `List` 반환 도구까지 같은 변경을 적용하지 않은 이유와, 향후 breaking change를 허용한다면 어떤 마이그레이션 전략을 쓰겠습니까?
